## API summary

`openshell` 0.0.47 is a **gRPC client SDK** for a running OpenShell gateway cluster ŌĆö it does NOT talk to Docker directly. The lifecycle is a sync context manager: `openshell.Sandbox(cluster=..., spec=openshell_pb2.SandboxSpec(...))` connects to a gateway via `SandboxClient.from_active_cluster()` (reads `~/.config/openshell/active_gateway` or `$OPENSHELL_GATEWAY`), calls `CreateSandbox`, waits for `SANDBOX_PHASE_READY`, then `DeleteSandbox` on exit. The image reference goes into `SandboxSpec.template.image` (a `str`). The sandbox policy is a `sandbox_pb2.SandboxPolicy` proto message (fields: `filesystem: FilesystemPolicy`, `landlock: LandlockPolicy`, `process: ProcessPolicy`, `network_policies: map<str, NetworkPolicyRule>`). There is no `sb.agent_endpoint` attribute; endpoint exposure requires a separate `ExposeService` gRPC call that returns `ServiceEndpointResponse.url`.

## Smoke result

**SMOKE FAIL ŌĆö API mismatch (expected, validates the spike's value).** Attempt #1 (speculative shape from the existing `OpenShellSandbox`): `openshell.Sandbox(image=..., policy=..., sandbox_id=...)` raises `TypeError: Sandbox.__init__() got an unexpected keyword argument 'image'`. Attempt #2 (real API): `SandboxClient.from_active_cluster()` raises `FileNotFoundError: ~/.config/openshell/active_gateway` ŌĆö no gateway cluster running locally (expected on a dev Mac; running the smoke against a real gateway is M1.4 work). Attempt #2b (proto construction only): `sandbox_pb2.SandboxPolicy(filesystem=..., landlock=LandlockPolicy(compatibility='best_effort'), network_policies={...})` constructs cleanly; `SandboxSpec.template.image` accepts the OCI ref; `Sandbox` public attributes are `['exec', 'exec_python', 'id', 'sandbox']` ŌĆö no `agent_endpoint`. Zero containers spawned (confirms OpenShell ŌēĀ Docker wrapper).

## Decision rationale

**Path A** ŌĆö Every binding-class row is OK. Row 8 (SMTP/process restriction) is satisfied at the network layer by deny-by-default `network_policies` (omit ports 25/587), which is sufficient for the M1.4 XPIA scenario. The remaining gaps ŌĆö rows 1, 2, 3, and 9 ŌĆö are all adapter-glue concerns handled inside `OpenShellSandbox.start()` in Gauntlet's own `sandbox.py`: constructor arg name (1), `SandboxSpec.template.image` threading (2), `NetworkPolicyRule` proto translation (3), and the post-`wait_ready` `ExposeService` call for `agent_endpoint` (9). M1.4 acceptance IS achievable against the already-installed `openshell==0.0.47` without an upstream PR.

## M1.4 work plan

Path A ŌĆö wire `OpenShellSandbox` against the already-installed `openshell==0.0.47` PyPI package. The M1.4 work plan is three concrete adapter-glue fixups inside `/src/gauntlet/sandbox.py`: (a) replace `openshell.Sandbox(image=..., policy=dict, sandbox_id=...)` with `openshell.Sandbox(spec=SandboxSpec(template=SandboxTemplate(image=agent_image), policy=<sandbox_pb2.SandboxPolicy proto>))`, (b) translate `SandboxPolicy.network_allow` list to a `NetworkPolicyRule` proto map in `SandboxPolicy.network_policies`, and (c) add a `SandboxClient._stub.ExposeService(...)` call after `wait_ready` to retrieve `ServiceEndpointResponse.url` as `agent_endpoint`. No upstream fork pin is required.

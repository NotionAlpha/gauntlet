# M1.3.6 — Lima-based OpenShell gateway setup

**Date:** 2026-05-24
**OpenShell fork SHA:** `686b24d` (upstream parity)
**Status:** Working setup; CI integration test gating added in M1.4.

## What this milestone delivers

A reproducible local OpenShell gateway running on **real Linux** inside a Lima-managed VM on macOS, with `gauntlet run` executable end-to-end against it. The host stays clean — no launchd service, no Homebrew gateway, no openshell CLI on the Mac. Everything lives inside the VM disk image at `~/.lima/openshell-gateway/`.

```
bash scripts/lima/gateway-up.sh                          # one-time, ~15-25 min
limactl shell openshell-gateway -- gauntlet --version    # works
```

## Why Lima, not Docker Desktop or Homebrew

OpenShell's enforcement primitives — **Landlock LSM** and **seccomp-bpf** — are Linux-kernel features. They simply don't exist on macOS, so a Mac-host gateway falls back to `best_effort` Landlock (the policy YAML still loads but the kernel doesn't enforce it). The M1.3.5 spike correctly classified this as a defense-in-depth observation, but for the M1.4 acceptance demo we want the report to mean what we'll claim it means: **real kernel-level deny** when an XPIA scenario tries forbidden egress.

Lima + Ubuntu 24.04 ARM64 gives us real Landlock + seccomp on Apple Silicon. The VM is one file on disk (`~/.lima/openshell-gateway/`) that can be reset, snapshotted, or deleted in seconds.

## Why build OpenShell from our fork

The NotionAlpha OSS AI Lab is positioned as a deep contributor to OpenShell (parent spec §6). The consistent identity story is "we run what we build." Today our fork (`NotionAlpha/OpenShell@686b24d`) is at upstream parity, so the bits are identical — but the build-from-fork workflow gives us the muscle to validate fork-only patches before opening upstream PRs.

There is also a **practical reason** that surfaced during this milestone:

## Material finding — upstream `openshell` Linux wheels are missing proto stubs

While provisioning, the broken wheel surfaced fast:

```
ImportError: cannot import name 'datamodel_pb2' from partially initialized
module 'openshell._proto' (most likely due to a circular import)
```

Inspection of the published wheels for `openshell==0.0.47` (and 0.0.46):

| Wheel | `_proto/__init__.py` | `*_pb2.py` files |
| --- | --- | --- |
| `macosx_13_0_arm64` | ✅ | ✅ all 4 |
| `manylinux_2_39_aarch64` | ✅ | **❌ all missing** |
| `manylinux_2_39_x86_64` | ✅ | **❌ all missing** |

The `_pb2.py` files (~290 KB total) are generated by `grpc_tools.protoc` and bundled in the macOS wheel; the Linux wheels are missing them entirely, which makes `openshell.sandbox` fail to import at the very first line. This is a real upstream packaging bug.

**Workaround for M1.3.6:** `scripts/lima/install-gauntlet-venv.sh` removes the broken PyPI install and pip-installs our fork's `openshell` package as editable, after running `mise run python:proto` against the fork's `proto/` sources to generate the missing stubs. The M1.3.5 spike couldn't see this bug because it ran on macOS, where the wheel happens to be complete — the Linux-substrate decision is already paying off.

**Upstream report:** see `docs/upstream-issues/openshell-linux-wheel-missing-protos.md` for the drafted issue (founder-submitted per the OSS Lab contribution convention).

## Setup walkthrough

### Prerequisites

- macOS (Apple Silicon supported; Intel works via x86_64 image selection in `lima/openshell-gateway.yaml`)
- Homebrew (`brew install lima` if not already installed)
- ~25 GB free disk for the VM image + Rust build cache

### One-command bootstrap

From the gauntlet repo root:

```bash
bash scripts/lima/gateway-up.sh
```

This:

1. Verifies (or installs) Lima.
2. Substitutes the host gauntlet repo path into a per-host copy of `lima/openshell-gateway.yaml` (writable mount).
3. `limactl create` + `limactl start` the VM (first boot ~5 min — downloads Ubuntu 24.04 ARM64 cloud image).
4. Inside the VM, runs `scripts/lima/install-openshell-from-fork.sh`:
   - apt-installs build deps (clang, libz3-dev, …).
   - Installs `mise` (build orchestrator).
   - Clones `NotionAlpha/OpenShell` at SHA `686b24d`.
   - `mise install` pulls Rust 1.95.0 + protoc 29.6 + sccache per the fork's `mise.toml`.
   - `mise run package:deb:install` builds the 3 binaries (openshell-cli, openshell-server, openshell-driver-vm), packages as `.deb`, `dpkg -i`, starts the `openshell-gateway.service` user-scope systemd unit, registers `https://127.0.0.1:17670` as the active gateway. **Cold cache: ~10–15 min.**
5. Inside the VM, runs `scripts/lima/install-gauntlet-venv.sh`:
   - Creates Python 3.12 venv at `~/work/gauntlet-venv`.
   - `pip install -e .[dev,integration]` from the mounted gauntlet repo.
   - `pip install -r agents/canonical/requirements.txt`.
   - Removes broken PyPI `openshell`, generates proto stubs via `mise run python:proto` in the fork, `pip install -e <fork>` to install our fork's openshell SDK.
6. Smoke checks: `openshell status: Connected`, `gauntlet --version` works.

### Daily lifecycle

```bash
limactl stop  openshell-gateway     # halt
limactl start openshell-gateway     # resume (gateway service auto-restarts)
limactl shell openshell-gateway     # interactive shell inside the VM
limactl delete openshell-gateway    # nuke (reclaim ~10GB of disk)
```

## Caveats / known issues

- **First-time provisioning takes 15–25 minutes** dominated by the cargo release build of OpenShell. Subsequent runs (script is idempotent) skip the rebuild.
- **VM resources hard-coded at 4 CPU / 8 GB RAM / 40 GB disk.** Edit `lima/openshell-gateway.yaml` for larger workloads.
- **HF_TOKEN must be passed at runtime** for the canonical agent's inference calls. Export it inside the VM session, or pass via `--env` to whatever sandbox-creation call sets the agent's environment.
- **The writable mount default is `~/code/gauntlet`** — `gateway-up.sh` substitutes the actual repo path automatically. If running `limactl start lima/openshell-gateway.yaml` directly (not via the script), edit the path.
- **macOS-only setup script.** Linux contributors can install Lima natively or run OpenShell directly without Lima.

## Relationship to M1.3.5 and M1.4

- **M1.3.5** (merged at `b77cdd2`) decided **Path A** — `openshell==0.0.47` API is sufficient; no upstream binding PR required. That decision still holds for the M1.4 adapter code. The bug we found here is a *packaging* bug, not an *API* bug.
- **M1.4** wires `OpenShellSandbox.start()` against the (now-working inside the VM) openshell Python SDK. M1.4 surfaced two additional gateway-setup findings worth recording here:
  - **Proto modules live at `openshell._proto.sandbox_pb2` + `openshell._proto.openshell_pb2`**, not `openshell.sandbox_pb2` (the path the M1.3.5 spike doc assumed). The fork-built SDK doesn't re-export the proto submodules at the top level. The adapter now uses `importlib.import_module` to load them.
  - **The gateway must bind to `0.0.0.0:17670`, not the default `127.0.0.1:17670`**, so sandbox containers can reach it from their Docker netns via the `host.openshell.internal:host-gateway` alias the OpenShell docker driver adds. `scripts/lima/install-openshell-from-fork.sh` now writes `OPENSHELL_BIND_ADDRESS=0.0.0.0` into `~/.config/openshell/gateway.env` and restarts the service. Without this override, sandbox containers fail to fetch their policy and enter `SANDBOX_PHASE_ERROR`.
